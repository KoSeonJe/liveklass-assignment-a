## 1. 개요

### 1.1 배경

크리에이터(강사)가 강의를 개설하고, 클래스메이트(수강생)가 강의에 신청·결제·취소할 수 있는 시스템을 구축한다. 정원 초과 신청은 거부되며, 결제 완료 시점에 수강이 확정된다. 결제 완료 후 일정 기간 내에만 수강 취소가 가능하다.

### 1.2 목표

본 과제는 실제 프로덕션 시스템의 완성도를 목표로 하지 않으며, 다음 세 가지 기술적 관점을 깊이 있게 검증하는 데 집중한다.

1. **동시성 제어 전략** — 마지막 한 자리 시나리오에서 정원이 깨지지 않으면서도 DB 락 직렬화를 피하는 설계.
2. **상태 전이 모델링** — Course, Enrollment, Payment 세 도메인의 상태 머신을 명시적으로 정의하고 잘못된 전이를 거부하는 구조.
3. **트랜잭션 경계 설계** — 인메모리 동시성 제어, DB 트랜잭션, 외부 API 호출이 섞이는 흐름에서 어디까지를 한 트랜잭션으로 묶을지 결정.

이 세 가지에 자원을 집중하기 위해 PENDING TTL, 중복 신청 방지, 다단계 보상 등 프로덕션 운영에 필요하지만 위 관점과 직접 관련 없는 항목은 의도적으로 다루지 않았다(§2.3 참고).

## 2. 환경 가정 및 제약사항

### 2.1 인프라 가정

- **단일 애플리케이션 인스턴스**: 다중 인스턴스 환경은 가정하지 않는다. 인메모리 자료구조 기반의 동시성 제어가 유효함을 전제로 한다.
- **단일 데이터베이스**: 단일 DB 인스턴스를 가정한다.
- **외부 결제 시스템**: 실제 결제 게이트웨이는 연동하지 않으며, Mock API로 대체한다.

### 2.2 명세 공백에 대한 결정

원본 요구사항이 명시하지 않은 사항에 대해 다음과 같이 결정하였다.

| # | 항목 | 결정 |
|---|---|---|
| 1 | 정원 카운팅 기준 | PENDING과 CONFIRMED 모두 정원을 점유한다. |
| 2 | 수강 취소 가능 기간 기준일 | CONFIRMED로 전이된 시점(결제 완료 시점)부터 7일 이내. |
| 3 | 강의 상태 전이 트리거 | 크리에이터의 명시적 요청 + 시스템 자동 전이(DRAFT → OPEN, `startDate` 도래 시 매일 00:00 KST). 그 외 자동 전이(CLOSED → OPEN 재개, OPEN → CLOSED 자동 마감 등)는 없다. |
| 4 | 인증·인가 | `X-User-Id` HTTP 헤더로 사용자를 식별한다. 별도 역할(Role) 검증 없이 모든 사용자가 모든 강의를 조회 가능하다. |
| 5 | 페이지네이션 방식 | offset/limit 기반(기본 size 20, max 100). |
| 6 | 결제 흐름 | 애플리케이션 내부 로직(상태 전이 등)을 모두 성공시킨 뒤 외부 결제 API를 호출하여, 외부 결제 취소 비용을 최소화한다. |
| 7 | 자동 전이 운영 가정 | 단일 인스턴스 운영을 전제로 한다. 분산 락(ShedLock 등) 미적용. 인메모리 `SeatCounter`는 전이 직후 일괄 초기화하여 정합성을 확보한다. 기준 타임존은 KST(`Asia/Seoul`)이다. |

### 2.3 의도적으로 다루지 않은 항목

다음 항목들은 과제 범위와 시간 제약을 고려하여 의도적으로 다루지 않았다. 미구현 사항은 README의 "미구현/제약사항"에 동일하게 명시한다.

| # | 항목 | 다루지 않은 이유 |
|---|---|---|
| 1 | PENDING 상태의 TTL(미결제 자동 취소) | 결제 흐름 단순화 및 과제 범위 한정을 위해 제외. PENDING은 사용자가 명시적으로 결제하거나 취소할 때까지 유지된다. |
| 2 | 동일 사용자의 동일 강의 중복 신청 검증 | 과제 범위 외로 판단. 동일 사용자가 같은 강의에 여러 Enrollment를 생성하는 것을 시스템 차원에서 막지 않는다. |
| 3 | CLOSED → OPEN 재전이 시 대기열 처리 정책 | 강의 상태 재개라는 엣지 케이스는 과제 범위 외로 판단. 대기열은 CLOSED 시점에 유지되나 자동 승급 로직은 정의하지 않는다. |
| 4 | 크리에이터의 강의 취소 (전체 수강생 일괄 환불·보상 트랜잭션 포함) | 다단계 외부 결제 취소·보상 트랜잭션 설계는 과제 범위 대비 복잡도가 크며, 평가 핵심(상태 전이/정원/동시성)과 직접 관련이 없어 제외. 크리에이터는 강의를 CLOSED로 전이시키는 것까지만 가능하다. |
| 5 | 실제 결제 게이트웨이 연동 | 요구사항에 따라 Mock으로 대체한다. |
| 6 | 본격적인 인증·인가 시스템(JWT, OAuth 등) | `X-User-Id` 헤더 기반의 단순 식별로 대체한다. |
| 7 | 강의 목록 조회의 상태/검색/정렬 필터 | 평가 핵심(상태 전이/정원/동시성)과 무관해 제외. `GET /api/courses`는 페이지네이션만 지원한다. |

## 3. 사용자 및 역할

### 3.1 크리에이터 (Creator)

강의를 개설하고 관리하는 사용자. 다음 행위를 수행할 수 있다.

- 강의 등록(DRAFT 상태로 생성)
- 강의 상태 변경 (DRAFT → OPEN, OPEN → CLOSED, CLOSED → OPEN 등)
- 자신의 강의에 대한 수강생 목록 조회

### 3.2 클래스메이트 (Classmate)

강의를 수강하는 사용자. 다음 행위를 수행할 수 있다.

- 강의 목록·상세 조회
- 수강 신청 (Enrollment 생성)
- 결제 진행
- 수강 취소 (조건부)
- 본인의 수강 신청 목록 조회

### 3.3 권한 모델

과제 단순화를 위해 사용자 역할을 별도 필드로 관리하지 않는다. 크리에이터 여부는 강의의 `creator_id`로 식별하며, **강의 상태 변경 및 강의별 수강생 목록 조회 시** `creator_id == X-User-Id` 검증을 수행한다. 강의 목록·상세 조회 및 본인 수강 신청 목록 조회는 모든 사용자에게 열려 있다.

## 5. 상태 머신

### 5.1 강의(Course) 상태 전이

```
            (크리에이터: 공개)            (크리에이터: 마감)
   DRAFT  ──────────────────▶   OPEN  ──────────────────▶   CLOSED
     │                            ▲                            │
     │                            │                            │
     │                            └────────────────────────────┘
     │                                 (크리에이터: 재공개)
     │
     │       (크리에이터: 즉시 마감)
     └────────────────────────────────────────────────────▶  CLOSED
```

| 전이 | 트리거 | 허용 여부 | 비고 |
|---|---|---|---|
| DRAFT → OPEN | ① 크리에이터의 공개 요청, ② 시스템 자동 전이(`startDate` 도래 시 매일 00:00 KST) | ✅ | 신청 가능 상태로 진입. 자동 전이는 단일 인스턴스 가정 하 분산 락 없이 동작. |
| DRAFT → CLOSED | 크리에이터의 즉시 마감 | ✅ | 초안 상태에서도 마감 가능 |
| OPEN → CLOSED | 크리에이터의 마감 요청 | ✅ | 신청 차단. 기존 신청은 유지. |
| CLOSED → OPEN | 크리에이터의 재공개 | ✅ | 신청 가능 상태로 복귀 |
| DRAFT → DRAFT 외 다른 상태 | — | ❌ | 정의된 전이만 허용 |

**상태별 신청 가능 여부**

| 상태 | 신청 가능 |
|---|---|
| DRAFT | ❌ |
| OPEN | ✅ |
| CLOSED | ❌ |

### 5.2 수강 신청(Enrollment) 상태 전이

```
   PENDING  ──────────▶  CONFIRMED  ──────────▶  CANCELLED
                            │
                            └──────────────────▶  PENDING (결제 실패 롤백)
```

| 전이 | 트리거 | 허용 여부 | 비고 |
|---|---|---|---|
| (없음) → PENDING | 수강 신청 | ✅ | 신규 생성 |
| PENDING → CONFIRMED | 결제 시도 시 내부적으로 전이 | ✅ | 외부 결제 API 호출 직전에 전이 |
| CONFIRMED → PENDING | 결제 실패 시 롤백 | ✅ | 외부 결제 API 응답이 실패인 경우 |
| CONFIRMED → CANCELLED | 수강 취소 요청 | ✅ | 결제 완료 후 7일 이내에만 가능 |
| PENDING → CANCELLED | — | ❌ | PENDING은 결제하거나 그대로 유지. 직접 취소 경로는 정의하지 않음 |
| CANCELLED → 다른 상태 | — | ❌ | 종단 상태. 재신청은 새 Enrollment 생성으로. |

**CONFIRMED → PENDING 전이의 의도**

결제 흐름은 "애플리케이션 내부 로직 모두 성공 → 외부 결제 API 호출"의 순서를 따른다. 이는 외부 결제가 성공한 뒤 애플리케이션 내부 상태 전이가 실패하여 결제를 취소해야 하는 비용을 회피하기 위함이다. 따라서 결제 API 호출 시점에는 이미 Enrollment가 CONFIRMED로 전이된 상태이며, 외부 결제가 실패하면 CONFIRMED → PENDING으로 롤백한다.

### 5.3 결제(Payment) 상태 전이

```
   PENDING  ───▶  SUCCESS  ───▶  CANCELLED (수강 취소 시 환불 처리)
      │
      └────────▶  FAILED
```

| 전이 | 트리거 | 허용 여부 | 비고 |
|---|---|---|---|
| (없음) → PENDING | 결제 진행 tx#1: Payment INSERT | ✅ | Payment row는 결제 시도 시점에 생성 |
| PENDING → SUCCESS | 외부 결제 Mock 성공 응답 | ✅ | |
| PENDING → FAILED | 외부 결제 Mock 실패 응답 | ✅ | Enrollment CONFIRMED → PENDING 롤백 동반 |
| SUCCESS → CANCELLED | 수강 취소(7일 이내) 환불 처리 | ✅ | |
| PENDING → CANCELLED | — | ❌ | 본 시스템에 트리거 부재. PENDING TTL·결제 중도 이탈은 §2.3 범위 외 |
| FAILED → CANCELLED | — | ❌ | 실패한 결제 환불 의미 없음 |
| CANCELLED → * | — | ❌ | 종단 상태 |

## 6. 비즈니스 규칙

### 6.1 정원 관리

- 강의 등록 시 정원(`max_capacity`)을 정수로 지정한다.
- 신규 수강 신청 시점에 정원이 차감되며, PENDING과 CONFIRMED 상태 모두 정원을 점유한다.
- 정원이 초과될 신청은 거부되며, 거부된 신청은 대기열로 진입한다(6.4 참고).

### 6.2 수강 신청

- 강의 상태가 OPEN인 경우에만 신청 가능하다.
- 신청 즉시 PENDING 상태의 Enrollment가 생성된다.
- PENDING은 결제 완료 또는 사용자의 명시적 행동(향후 결제) 이전까지 유지된다.

### 6.3 결제 및 수강 확정

- 결제 요청 시 다음 순서로 처리된다.
  1. Enrollment를 PENDING → CONFIRMED로 전이
  2. Payment 레코드를 PENDING 상태로 생성 (멱등키 포함)
  3. 외부 결제 Mock API 호출
  4. 응답에 따라 Payment 상태를 SUCCESS 또는 FAILED로 업데이트
  5. FAILED인 경우 Enrollment를 CONFIRMED → PENDING으로 롤백
- 멱등키를 통해 동일 결제의 중복 호출을 방지한다.

### 6.4 대기열

- 강의 신청 시 정원이 모두 점유되어 있으면 대기열로 진입한다.
- 대기열은 강의별로 분리되며, FIFO 순서를 보장한다.
- 다음 사건이 발생하면 대기열의 최선두 사용자가 자동으로 PENDING 상태의 Enrollment로 승급된다.
  - 다른 클래스메이트가 수강 취소(CONFIRMED → CANCELLED)
  - 결제 실패로 인한 정원 반환 후 보상 처리 시
- 단일 인스턴스를 가정하므로 대기열은 인메모리로 관리한다. 서버 재기동 시 대기열 데이터는 손실된다(제약사항으로 명시).

### 6.5 수강 취소

- CONFIRMED 상태인 Enrollment에 한해 취소 가능하다.
- 취소 가능 기간은 CONFIRMED로 전이된 시점부터 7일 이내이다.
- 7일 경과 후 취소 요청은 거부된다.
- 취소 시 다음이 연쇄적으로 처리된다.
  1. Enrollment 상태를 CANCELLED로 전이
  2. 정원 1 반환(인메모리 카운터 및 DB 동기)
  3. 대기열에서 최선두 사용자를 PENDING으로 승급
  4. Payment를 CANCELLED로 전이(환불 처리, Mock)

## 7. 주요 유스케이스

### 7.1 강의 등록 및 공개

1. 크리에이터가 강의를 등록한다 → DRAFT 상태 생성
2. 크리에이터가 강의를 공개한다 → DRAFT → OPEN 전이
3. 클래스메이트가 강의 목록에서 해당 강의를 확인할 수 있다.

### 7.2 정상 수강 신청 및 결제 확정

1. 클래스메이트가 OPEN 상태의 강의에 신청한다.
2. 정원이 차감되고 PENDING 상태의 Enrollment가 생성된다.
3. 클래스메이트가 결제를 진행한다.
4. Enrollment가 CONFIRMED로 전이되고 외부 결제 API가 호출된다.
5. 결제 성공 응답 시 Payment 상태가 SUCCESS가 된다.

### 7.3 정원 만석 시 대기열 진입

1. OPEN 강의의 정원이 모두 점유된 상태에서 신청이 들어온다.
2. 인메모리 카운터에서 차감 시도가 실패한다.
3. 신청자는 대기열에 등록되며, 클라이언트에는 "대기열 등록" 응답이 반환된다.

### 7.4 수강 취소와 대기자 승급

1. CONFIRMED 상태의 클래스메이트가 결제 후 5일 째에 취소를 요청한다.
2. Enrollment가 CANCELLED로 전이된다.
3. 정원이 1 반환되며, 대기열의 최선두 사용자가 PENDING Enrollment로 승급된다.
4. 승급된 사용자는 결제 진행이 가능하다.

### 7.5 결제 실패와 상태 롤백

1. 클래스메이트가 결제를 시도한다.
2. Enrollment가 PENDING → CONFIRMED로 전이된다.
3. 외부 결제 API 호출이 실패 응답을 반환한다.
4. Payment가 FAILED로 기록되고, Enrollment는 CONFIRMED → PENDING으로 롤백된다.

## 8. API 엔드포인트 목록

요청·응답 상세 스펙은 Tech-spec을 참고한다. 본 PRD는 외부 계약 수준의 목록만 제공한다.

### 8.1 강의(Course)

| Method | Path | 설명 |
|---|---|---|
| POST | `/api/courses` | 강의 등록 (DRAFT 생성) |
| GET | `/api/courses` | 강의 목록 조회 (페이지네이션) |
| GET | `/api/courses/{courseId}` | 강의 상세 조회 (현재 신청 인원 포함) |
| PATCH | `/api/courses/{courseId}/status` | 강의 상태 전이 |
| GET | `/api/courses/{courseId}/enrollments` | 강의별 수강생 목록 조회 (크리에이터 전용) |

### 8.2 수강 신청(Enrollment)

| Method | Path | 설명 | 성공 응답 |
|---|---|---|---|
| POST | `/api/courses/{courseId}/enrollments` | 수강 신청 | **201 Created** (자리 확보 → PENDING Enrollment 생성) / **202 Accepted** (정원 만석 → 대기열 등록) |
| POST | `/api/enrollments/{enrollmentId}/payment` | 결제 진행 | 200 OK |
| POST | `/api/enrollments/{enrollmentId}/cancel` | 수강 취소 | 200 OK |
| GET | `/api/me/enrollments` | 내 수강 신청 목록 조회 (페이지네이션) | 200 OK |

**201/202 구분 의도**: 201은 Enrollment 리소스가 실제로 생성된 경우, 202는 요청 수락은 됐으나 Enrollment 미생성(대기열 인메모리 등록 상태)인 경우. 응답 body·상세 형식은 Tech-spec §9.2 참고. 대기 사용자는 응답 받은 뒤 떠나도 되며, 본인 신청 목록 조회로 상태 확인. 결제 가능 시점의 능동 통지는 범위 외(§2.3).

### 8.3 공통 헤더

| 헤더 | 설명 | 필수 여부 |
|---|---|---|
| `X-User-Id` | 요청 사용자의 ID | 모든 요청에 필수 |

## 9. 비기능 요구사항

### 9.1 동시성

- 동시에 N명이 마지막 한 자리에 신청하는 상황에서, 정확히 1명만 PENDING 상태의 Enrollment를 획득해야 한다.
- 나머지 N-1명은 대기열로 진입한다.
- 정원 초과는 어떤 상황에서도 발생하지 않아야 한다.

### 9.2 일관성

- 인메모리 카운터와 DB의 현재 신청 인원은 논리적으로 동기화된다.
- DB 갱신이 실패하는 경우 인메모리 카운터를 즉시 복원하여 정합성을 보장한다.
- 자세한 보장 메커니즘은 Tech-spec 참고.

### 9.3 페이지네이션

- 강의 목록 및 수강 신청 목록은 offset/limit 페이지네이션을 지원한다.
- 기본 size 20, 최대 size 100.

## 10. 평가 관점 대응

§1.2에서 명시한 세 가지 핵심 관점에 본 시스템이 어떻게 대응하는지 요약한다.

| 평가 관점 | 본 시스템의 대응 | 상세 위치 |
|---|---|---|
| 동시성 제어 전략 | 락 기반 직렬화가 아닌, 인메모리 `AtomicInteger` 카운터와 DB 원자적 UPDATE의 조합을 채택. 정원 만석은 인메모리에서 fast-fail로 판별되어 DB까지 도달하지 않으며, 거부된 신청은 대기열로 라우팅된다. DB 갱신 실패 시 인메모리 카운터를 복원하는 보상 로직으로 정합성을 보장한다. | Tech-spec §4, Appendix A |
| 상태 전이 모델링 | Course(DRAFT/OPEN/CLOSED), Enrollment(PENDING/CONFIRMED/CANCELLED), Payment(PENDING/SUCCESS/FAILED/CANCELLED) 세 도메인의 상태 머신을 Enum과 전이 매핑으로 명시적으로 정의한다. 모든 전이는 도메인 객체의 메서드를 통해서만 가능하며, 허용되지 않은 전이는 예외로 거부한다. | PRD §5, Tech-spec §7 |
| 트랜잭션 경계 설계 | Facade-Service 2계층으로 트랜잭션 외부의 동시성 제어와 트랜잭션 내부의 DB 조작을 분리한다. 결제 흐름에서 외부 API 호출은 트랜잭션 외부로 빼서 커넥션 점유 시간을 최소화하며, 결제 실패 시 Enrollment를 PENDING으로 롤백하는 보상 로직을 Facade에 위치시킨다. | Tech-spec §2.2, §6 |
